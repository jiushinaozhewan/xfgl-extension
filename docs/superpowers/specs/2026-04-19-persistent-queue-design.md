# 持久化待处理队列设计

## 背景

当前扩展会在后台维护 `dataList` 和 `currentIndex`，但侧边栏里的“数据列表”不会随着处理结果同步变化。流程中断、侧边栏关闭、扩展重载后，用户无法直观看到剩余待处理项，也不方便继续执行。

## 目标

- 成功处理的条目立即从“待处理列表”移除
- 失败或无数据的条目从待处理列表移出，并沉淀到单独的异常列表
- 任务状态持久化，扩展重启或侧边栏重开后仍可继续
- 保持主流程自动化逻辑尽量不变，避免扩大改动范围

## 方案

后台状态改为显式队列：

- `pendingItems`：尚未成功处理的数据
- `successItems`：成功处理的数据
- `retryItems`：失败或无数据的数据

处理规则：

- 开始新任务时，文本框解析结果写入 `pendingItems`
- 成功时，从 `pendingItems` 头部移除当前项，并追加到 `successItems`
- 失败或无数据时，从 `pendingItems` 头部移除当前项，并追加到 `retryItems`

持久化规则：

- 使用 `chrome.storage.local` 保存完整状态
- 启动或重新打开侧边栏时恢复队列状态
- 若存在历史 `pendingItems`，用户可直接继续任务

界面规则：

- 原“数据列表”文本框在运行/恢复时展示当前 `pendingItems`
- 新增“失败/无数据列表”只读文本框，展示 `retryItems`
- 成功、失败、待处理统计分别对应三组显式队列

## 边界

- 本次不实现失败列表的一键重试
- 本次不改动自动化页面交互协议
- 对重复数据按“逐条消费”处理，不做去重
